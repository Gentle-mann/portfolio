Crisis Memory Bridge

An AI-powered memory layer for crisis counseling hotlines. When a caller reaches out again and gets a different volunteer, the new volunteer already has full context. The caller never has to re-explain their worst moment to a stranger.

Role: Solo developer Timeline: 1 day (hackathon) Stack: FastAPI, Claude API, memU, SSE Status: Live on Railway

The Problem

Japan has one of the highest suicide rates among developed nations. In 2022, over 21,000 people died by suicide — roughly 58 people per day. Crisis hotlines are a critical lifeline, but they share a structural problem that no amount of volunteer training can solve.

Inochi no Denwa — Japan's oldest crisis hotline — operates with 7,000 volunteers handling 700,000 calls per year. Yorisoi Hotline provides 24-hour multilingual support. TELL Lifeline serves the English-speaking community. These organizations do extraordinary work.

But when a caller reaches out again, they almost certainly get a different volunteer. That volunteer has zero context. The caller must re-explain their situation, their triggers, what was tried before, what helped — from scratch.

No one should have to retell their trauma because the shift changed.

Why This Matters

  • Re-traumatization: Asking someone in crisis to retell their worst moment to a stranger is itself harmful. Forced narrative repetition can intensify trauma responses.
  • No institutional memory: Individual volunteers may remember a caller, but the organization doesn't. Each interaction starts at zero.
  • Volunteer burnout: Without context, volunteers spend time in assessment mode rather than support mode — exhausting for both parties.
  • Missed escalation: A caller's risk may be trending upward across sessions, but if each session is independent, no one sees the trajectory.

The Solution

Crisis Memory Bridge adds an AI-powered memory layer between sessions. It observes conversations, extracts structured context in real-time, and surfaces that context to the next volunteer — automatically.

  1. Volunteer A talks with a caller. The AI observes and extracts context in real-time — triggers, mood, strategies, risk level — into a live sidebar.
  2. Session ends. The AI extracts structured memories and stores them.
  3. Volunteer B connects with the same caller. Before the conversation starts, they see a full briefing, memory timeline, and changes since the last session.
  4. The caller feels known. They don't have to start over.

Architecture

Frontend (Vanilla HTML/CSS/JS) │ Setup Screen → Chat → Extraction → Reset │ SSE streaming, timeline rendering, live context │ ├── HTTP + Server-Sent Events │ Backend (FastAPI + Python) ├── LLM Service (Claude API) │ 8 distinct operations, 3 run in parallel │ ├── Memory Store (Abstract) │ ├── LocalMemoryStore → JSON on disk (primary) │ └── HybridMemoryStore → Local + memU (supplementary) │ └── SSE Streaming token → stream_end → done (context + coaching)

LLM Orchestration

Every LLM call routes through a central service with 8 distinct operations. The key insight: live context extraction, volunteer coaching, and reply suggestions run in parallel after each message exchange — 3 concurrent LLM calls, zero added latency.

The streaming architecture splits into two events: stream_end (re-enables volunteer input immediately) and done (delivers context/coaching 2-4 seconds later). The volunteer can start composing their next message while background analysis completes.

Hybrid Memory

Memory is stored in two complementary layers:

  • Local structured store — Domain-specific schema (triggers, risk level, safety plan, warnings, strategies) as JSON. Powers the timeline, diffs, and briefings. Always the primary system.
  • memU semantic layer — Receives raw conversations and provides supplementary context that structured extraction missed (personal details, emotional patterns, offhand mentions). Optional — app works identically without it.

On handoff, the new volunteer sees "Changes Since Last Session" — color-coded items showing new triggers (yellow), risk escalations (red), and newly effective strategies (green). This is the single most valuable piece of information for a handoff.

Key Features

Live Context Extraction

Sidebar updates in real-time as the conversation progresses — triggers, mood, risk level, effective strategies.

Volunteer Coaching

Each message scored with technique identification and feedback. Color-coded inline hints (good/needs improvement/caution).

Risk Escalation Alerts

Flashing red banner when risk increases. Emergency protocol overlay with safety checklist at high risk.

Smart Reply Suggestions

Context-aware openers for returning callers. LLM-generated follow-ups after each exchange. Click to populate input.

Session Timeline & Diffs

Visual timeline across sessions with cross-session change detection. NEW, ESCALATED, and WORKED items highlighted.

Full EN/JA Localization

Every string, LLM output, voice input, and system message in both English and Japanese. 75+ translation keys.

Supervisor Dashboard

Multi-caller grid, risk trend charts, session replay with timed playback, and per-caller analytics.

Voice Input & Sound Alerts

Web Speech API for hands-free dictation. Web Audio API oscillator tones for risk alerts and background notifications.

Beyond Crisis Counseling

The memory-layer-for-handoff pattern applies wherever humans are passed between helpers:

  • Healthcare: Nursing shift handoffs, primary care continuity, ED triage for returning patients
  • Social services: Caseworker turnover, refugee resettlement, homeless outreach
  • Education: IEP tracking across teachers, school counselor transitions, tutoring platforms
  • Legal: Public defenders with high caseloads, immigration legal aid spanning years
The volunteer changes. The context doesn't.
